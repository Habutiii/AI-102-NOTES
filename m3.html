Azure AI Language

Key Takeaways:
  • Sentiment analysis with opinion mining for fine-grained sentiment extraction
  • Entity recognition (NER) and entity linking to knowledge bases
  • Language detection and key phrase extraction
  • PII detection to identify personally identifiable information in text
  • Use Azure AI Translator for machine translation with custom models

Overview

Azure AI Language is the unified NLP service providing pre-built and custom language capabilities through a single endpoint. All features are accessed via the /language/:analyze-text REST endpoint or the azure-ai-textanalytics SDK.

Single endpoint, many features All pre-built features use the same base URL: POST /language/:analyze-text?api-version=2023-04-01 with a "kind" field in the body selecting the feature.

Pre-built Features

Sentiment Analysis

Document-level and sentence-level sentiment: positive, neutral, negative, mixed with confidence scores.

Opinion Mining

Aspect-based sentiment. Extracts target aspects and their opinions. E.g., "room: positive, service: negative". Enable with "opinionMining": true.

Key Phrase Extraction

Returns main talking points from text. No training needed. Useful for summarising large volumes of text.

Language Detection

Returns ISO 639-1 language code + confidence score. Returns (Unknown) for mixed-language or ambiguous text.

Named Entity Recognition (NER)

Extracts typed entities: Person, Location, Organization, Date, Quantity, URL, Email, IP, Phone, etc.

Entity Linking

Disambiguates entities by linking to Wikipedia. E.g., "Mars" → Mars (planet) vs. Mars (candy bar) with confidence score.

PII Detection

Identifies and optionally redacts Personally Identifiable Information: SSNs, credit card numbers, emails, phone numbers, addresses.

Text Summarization

Two modes: extractive (selects key sentences verbatim) and abstractive (generates new summary text). Both are async operations.

REST API — Sentiment Analysis

POST https://<endpoint>.cognitiveservices.azure.com/language/:analyze-text?api-version=2023-04-01

{
  "kind": "SentimentAnalysis",
  "analysisInput": {
    "documents": [
      { "id": "1", "language": "en", "text": "I love Azure AI services!" }
    ]
  }
}

Text Summarization

Both summarization types use an async long-running operation pattern (submit → poll → retrieve).

TypeHow It WorksOutput
ExtractiveScores each sentence by importance and selects the top N. Sentences are verbatim from the original text.List of sentences with rank score, offset, length
AbstractiveGenerates new summary text using a generative model. May not match exact phrases in the source.Generated summary paragraph(s)
from azure.ai.textanalytics import TextAnalyticsClient
from azure.core.credentials import AzureKeyCredential

client = TextAnalyticsClient(endpoint, AzureKeyCredential(key))

# Extractive summarization — returns key sentences verbatim
poller = client.begin_extract_summary(
    ["Long article text..."],
    max_sentence_count=3
)
for doc in poller.result():
    for sentence in doc.sentences:
        print(sentence.text, f"(rank: {sentence.rank_score:.2f})")

# Abstractive summarization — generates new summary text
poller = client.begin_abstract_summary(
    ["Long article text..."],
    sentence_count=2
)
for doc in poller.result():
    print(doc.summaries[0].text)

Custom Text Classification

Single-label Classification

Each document gets exactly one class. Use when categories are mutually exclusive (e.g., Sports / Politics / Technology).

Multi-label Classification

A document can belong to multiple classes. Use when overlap is possible (e.g., a film that is both Action and Comedy).

Workflow — Custom Features (Classification, NER, Q&A) Language Studio → Create project → Upload & label data → Train → Evaluate (precision / recall / F1) → Deploy → Call prediction endpoint

Custom Named Entity Recognition

Extend pre-built NER with domain-specific entity types. Examples: part numbers, medical procedure codes, legal clause references.

  • Minimum: ~10 labeled examples per entity type
  • Recommended: 50–200 examples for production accuracy
  • Labels are applied at the character span level in Language Studio

Question Answering

Builds a knowledge base (KB) from FAQs, documents, or URLs and answers natural language questions.

Custom Question Answering

Build KB from documents, URLs, or Q&A pairs. Supports multi-turn conversations (follow-up prompts, up to 5 per answer). Requires Azure AI Language resource; uses Azure AI Search for large KBs.

Pre-built Q&A

Query a passage of text directly — no KB, no training. One-shot extraction. Useful for reading comprehension tasks.

Exam Tip — Q&A vs CLU Q&A: FAQ-style knowledge retrieval ("What are your opening hours?"). CLU: Intent understanding to trigger app actions ("Book me a flight to Paris next Friday"). If the question has a document/KB answer → Q&A. If you need to extract slots and route to an action → CLU.

Active Learning

Active learning suggests new Q&A pairs based on user queries that did not match any existing answer in the knowledge base. The service collects real traffic queries and surfaces them as suggestions once a minimum traffic threshold is reached.

  • To enable: open the project in Language Studio → Settings and set Active learning to Enabled.
  • Active learning is OFF by default — you must explicitly turn it on.
  • If no suggestions appear after enabling, it means not enough queries have been logged yet — wait for more traffic.
Exam Tip — Active Learning Active learning is OFF by default. You must explicitly enable it in project settings before any suggestions will appear. If you see no suggestions, check that active learning is enabled first before assuming there is a traffic problem.

FAQ Import — What Gets Extracted

When importing an FAQ document or URL into a Custom Question Answering knowledge base, the service extracts only Question/Answer text pairs. It does not extract images, tables, or text formatting — only plain text Q&A content is ingested.

Synonym API Format

Synonyms are defined using the "alterations" array in the API. Each alteration string must be a plain phrase with no special characters.

Invalid synonym strings: strings that start with #, &, <, or > are invalid and will cause the API call to fail. For example, "#troubleshoot" is not a valid alteration value. 
{
  "synonyms": [
    {
      "alterations": [
        "fix problems",
        "troubleshoot",
        "resolve issues"
      ]
    }
  ]
}

Question Answering — Costs

Costs for Custom Question Answering depend on:

  • Number of transactions (API calls made to the prediction endpoint)
  • Hosting tier of the Azure AI Language resource (Free F0 vs Standard S)
  • Text record volume and associated Azure AI Search storage for large knowledge bases

Suitable Scenarios for Question Answering

Good fit
  • Structured FAQ-style documents
  • Customer support knowledge bases
  • HR policy or internal helpdesk chatbots
Not suited for
  • Open-domain text generation
  • Complex multi-step reasoning tasks
  • Real-time data lookup or live information retrieval

PII Detection

The PII Detection feature identifies and can redact Personally Identifiable Information from text. It is available as a pre-built task in Azure AI Language with no training required.

PII Entity Categories

CategoryExamples
PersonIndividual names (e.g., "John Smith")
AddressStreet addresses, postal codes
PhoneNumberMobile and landline numbers
EmailEmail addresses
IPAddressIPv4 and IPv6 addresses
CreditCardNumberPayment card numbers
SSNSocial Security Numbers (US)
PassportPassport numbers
DriversLicenseDriving licence numbers
Exam Tips — PII Categories
  • To flag staff names in documents → use the "Person" category.
  • To remove both names and addresses from resumes → specify "Person" and "Address" in the categories parameter.

REST API — PII Entity Recognition

POST https://<endpoint>.cognitiveservices.azure.com/language/:analyze-text?api-version=2023-04-01

{
  "kind": "PiiEntityRecognition",
  "analysisInput": {
    "documents": [
      { "id": "1", "language": "en", "text": "John Smith lives at 42 Main Street." }
    ]
  },
  "parameters": {
    "piiCategories": ["Person", "Address"],
    "redactionCharacter": "*"
  }
}

Key Phrase Extraction vs Summarization

These three features are often confused in exam questions. Choose the right one based on what the output should be.

FeatureOutputTask KindUse When
Key Phrase Extraction List of important words and short phrases from the document KeyPhraseExtraction You need key concepts or topic tags — "summarize into key phrases"
Extractive Summarization The most important full sentences copied verbatim from the document ExtractiveSummarization You want a condensed version using the original sentences
Abstractive Summarization Newly generated summary sentences (paraphrased, not copied) AbstractiveSummarization You want a fluent summary that may not match source text verbatim
Exam Tip — Q21 Pattern If an exam question asks you to "summarize text documents into key phrases", the answer is Key Phrase Extraction, not Extractive or Abstractive Summarization. Key Phrase Extraction specifically returns phrases; the summarization tasks return sentences or paragraphs.
Summarization is async: Both extractive and abstractive summarization use a long-running operation pattern (submit → poll → retrieve results). Key Phrase Extraction is synchronous.
