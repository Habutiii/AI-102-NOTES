Exam Overview

What is AI-102?

The Azure AI Engineer Associate certification validates your ability to design, build, and deploy AI solutions using Azure Cognitive Services, Azure Machine Learning, and other Azure AI tools. Candidates typically have experience with C#, Python, or JavaScript and familiarity with Azure fundamentals.

Azure AI Services — Plan & Manage

What are Azure AI Services?

Azure AI Services (formerly Cognitive Services) are cloud-based REST APIs and SDKs that allow you to add AI capabilities to applications without needing ML expertise. They are organised into categories: Vision, Speech, Language, Decision, and Azure OpenAI.

Single-Service Resource
  • One endpoint per service (e.g., Speech only)
  • Dedicated billing & quota per service
  • Use when you need one specific AI service
Multi-Service Resource
  • Single endpoint for multiple services
  • Single key authenticates all services
  • Consolidated billing — easier to manage
  • Resource type: CognitiveServices

Authentication

Azure AI Services support two primary authentication methods:

Subscription Key (API Key)

Passed as Ocp-Apim-Subscription-Key header. Two keys provided — rotate without downtime.

Microsoft Entra ID (AAD)

Use service principals or managed identities. Preferred for production — no secrets in code.

Token-Based Auth

Exchange key for a 10-minute JWT token at /sts/v1.0/issueToken. Pass in Authorization: Bearer {token}.

// Authenticate with API Key
var client = new TextAnalyticsClient(
    new Uri("https://<endpoint>.cognitiveservices.azure.com"),
    new AzureKeyCredential("<api-key>")
);

// Authenticate with Managed Identity
var credential = new DefaultAzureCredential();
var client2 = new TextAnalyticsClient(endpoint, credential);

Securing AI Services

  • Azure Key Vault — store API keys; grant access via managed identities, never hardcode secrets
  • Network restrictions — limit access by virtual network or IP allowlists in the resource Networking tab
  • Private Endpoints — traffic stays within Azure backbone; disable public access
  • RBAC — assign Cognitive Services User to allow API calls; Cognitive Services Contributor to manage the resource
  • Diagnostic logging — send to Log Analytics; query with KQL
Best Practice Always use Managed Identity + Key Vault in production. For local dev, use DefaultAzureCredential which automatically picks up az login credentials.

Computer Vision

Azure AI Vision (Image Analysis)

The Azure AI Vision service (v4.0 Image Analysis API) provides a unified endpoint for analysing images. It replaces earlier Computer Vision v3 endpoints.

Caption & Dense Captions

Generates human-readable description of an image and per-region captions. Uses Florence foundation model.

Object Detection

Detects objects and returns bounding boxes with labels and confidence scores.

Smart Crop

Returns cropping coordinates that preserve the most visually interesting region.

Tags

Returns list of content tags with confidence scores (e.g., "outdoor", "sky", "person").

People Detection

Detects people in an image (bounding boxes). Does NOT identify individuals.

Background Removal

Segments the subject from background. Returns PNG with alpha channel.

API Call (v4.0)

POST https://<endpoint>/computervision/imageanalysis:analyze
    ?api-version=2023-02-01-preview
    &features=Caption,Tags,Objects
    &language=en

Content-Type: application/json
Ocp-Apim-Subscription-Key: <key>

{ "url": "https://example.com/image.jpg" }

Custom Vision

Azure AI Custom Vision lets you train models for image classification and object detection on your own dataset without coding.

Image Classification

Assigns one or more labels to an entire image. Types: Multiclass (one label) or Multilabel (multiple labels). Min: 5 images per tag to train.

Object Detection

Identifies object locations with bounding boxes. Min: 15 images per tag. Labels must be drawn per image.

Exam Tip Custom Vision uses two separate resources: a Training resource and a Prediction resource. Training key is used to create/train; Prediction key is used for inference calls. You export models as ONNX, TensorFlow, CoreML, or Docker container for offline use.
ConceptDetail
IterationEach training run creates an iteration. Publish an iteration to make it available for prediction.
DomainCompact domains (faster, exportable) vs. General domains (more accurate, cloud only)
Quick TestTest an unpublished iteration within the portal
Performance metricsPrecision, Recall, AP (Average Precision) per tag

Face API

The Face API detects, analyzes, and compares human faces.

Responsible AI Note Face identification, verification, and emotion detection features require Limited Access approval from Microsoft. Default capability is face detection only (locations, attributes).
Face Detection

Returns face rectangles, age estimate, gender, emotion, accessories, blur, exposure, noise, occlusion, headPose, smile, glasses.

Face Verification

Compares two faces — are they the same person? Returns confidence 0–1.

Face Identification

Matches a face against a PersonGroup of known individuals. Requires training a PersonGroup first.

Find Similar

Finds similar faces from a FaceList without a PersonGroup.

Face Identification Workflow

1. Create PersonGroup → 2. Add Persons → 3. Add Face images → 4. Train PersonGroup
5. Detect face in new image (get faceId) → 6. Identify faceId against PersonGroup

Natural Language Processing (Azure AI Language)

Azure AI Language Overview

Azure AI Language (unified service, formerly Text Analytics + LUIS + QnA Maker) provides pre-built and custom NLP capabilities through a single resource.

Sentiment Analysis

Returns document-level and sentence-level sentiment: positive, neutral, negative, mixed. Also returns confidence scores for each.

Key Phrase Extraction

Identifies the main talking points in text. No training needed. Returns a list of phrases.

Language Detection

Identifies the language of input text. Returns ISO 639-1 code + confidence. Handles mixed-language as "unknown".

Named Entity Recognition (NER)

Extracts entities: Person, Location, Organization, Date, Quantity, Email, URL, Phone, IP address, etc. with confidence scores.

Entity Linking

Disambiguates entities and links to Wikipedia. E.g., "Mars" → planet vs. candy bar.

PII Detection

Identifies and can redact Personally Identifiable Information (PII) like SSNs, credit card numbers, emails, phone numbers.

REST API Example — Sentiment

POST https://<endpoint>.cognitiveservices.azure.com/language/:analyze-text?api-version=2023-04-01

{
  "kind": "SentimentAnalysis",
  "analysisInput": {
    "documents": [
      { "id": "1", "language": "en", "text": "I love Azure AI services!" }
    ]
  }
}

Opinion Mining (Aspect-Based Sentiment)

Extension of Sentiment Analysis. Extracts aspects (targets) and their opinions (sentiment) from text.

Example: "The room was clean but the service was slow" → room: positive, service: negative

Enable by adding "opinionMining": true to the Sentiment Analysis request.

Custom Text Classification

Single-label Classification

Each document gets exactly one class. Use when categories are mutually exclusive (e.g., news category: Sports / Politics / Tech).

Multi-label Classification

A document can belong to multiple classes simultaneously. Use when overlap is possible (e.g., a movie is both Action AND Comedy).

Workflow Create project in Language Studio → Upload/label data → Train model → Evaluate → Deploy → Call prediction endpoint

Custom Named Entity Recognition (NER)

Train a model to extract domain-specific entities not covered by pre-built NER. Examples: part numbers, medical codes, contract clauses.

Minimum: ~10 labeled examples per entity type; recommended 50–200 for good accuracy.

Question Answering

Creates a knowledge base from FAQs, documents, or URLs. Answers natural language questions.

Custom Question Answering

Build KB from documents, URLs, Q&A pairs. Supports follow-up prompts (multi-turn). Requires Azure AI Language resource + optional AI Search for larger KBs.

Pre-built Q&A

Query a passage of text directly without a KB. One-shot extraction. No training needed.

Exam Tip — QnA vs CLU Q&A: Best for FAQ-style knowledge base answering ("What are your hours?"). CLU: Best for understanding user intent and extracting entities to trigger actions ("Book a flight to Paris next Friday").

Multi-Turn Conversations (Follow-up Prompts)

You can chain answers with follow-up prompts to guide users through a conversation tree. Each answer can have up to 5 follow-up prompts.

Azure AI Speech

Speech Service Capabilities

Speech-to-Text (STT)

Converts spoken audio to text. Supports real-time and batch transcription. Can use custom acoustic/language models.

Text-to-Speech (TTS)

Converts text to natural-sounding audio. 400+ voices, 140+ languages. Supports SSML for customization.

Speech Translation

Real-time speech translation — transcribes and translates simultaneously. Uses Speech SDK.

Speaker Recognition

Identifies or verifies speakers by voice. Speaker Verification (1:1) and Speaker Identification (1:N).

Intent Recognition

Combines STT with CLU to extract intent and entities from speech in one step.

Custom Neural Voice

Train a custom TTS voice model from voice samples. Requires Limited Access approval.

Speech SDK Quickstart Pattern

import azure.cognitiveservices.speech as speechsdk

config = speechsdk.SpeechConfig(subscription=key, region=region)
config.speech_recognition_language = "en-US"

# From microphone
audio_config = speechsdk.AudioConfig(use_default_microphone=True)
recognizer = speechsdk.SpeechRecognizer(speech_config=config, audio_config=audio_config)
result = recognizer.recognize_once_async().get()

if result.reason == speechsdk.ResultReason.RecognizedSpeech:
    print(f"Recognized: {result.text}")

SSML — Speech Synthesis Markup Language

SSML lets you control pronunciation, speaking rate, pitch, pauses, and voice selection in TTS.

<speak version="1.0" xmlns="http://www.w3.org/2001/10/synthesis" xml:lang="en-US">
  <voice name="en-US-JennyNeural">
    <prosody rate="slow" pitch="+5%">
      Welcome to <emphasis level="strong">Azure AI</emphasis> Speech services.
    </prosody>
    <break time="500ms"/>
    How can I help you today?
  </voice>
</speak>

Custom Speech

Improve STT accuracy for domain-specific terms, accents, and environments by providing custom training data:

  • Audio + Human-labeled transcripts — most accurate adaptation, needs 1–20 hrs audio
  • Plain text — improves language model (words, phrases, n-grams)
  • Pronunciation — add custom pronunciations for unusual words

Batch Transcription

Transcribe large audio files asynchronously. Submit via REST API, audio must be in Azure Blob Storage.

POST /speechtotext/v3.1/transcriptions
{
  "contentUrls": ["https://storage.blob.core.windows.net/.../audio.wav"],

Language Understanding — CLU (Conversational Language Understanding)

CLU Overview

CLU (successor to LUIS) understands the intent behind natural language utterances and extracts entities. Used in chatbots, voice commands, and virtual assistants.

Intent

The goal or purpose of a user's utterance. Example: "Book a flight to Paris" → intent = BookFlight. Always include a None intent for utterances outside the app's scope.

Entity

Key information extracted from the utterance. Example: "Paris" → Destination entity. Types: Machine-learned, List, Regex, Prebuilt.

Utterance

A sample user input used to train the model. More diverse utterances per intent = better model. Minimum ~5 per intent; aim for 15–30+.

Entity Types

Entity TypeWhen to UseExample
Machine-learnedFlexible entities from contextProduct names, descriptions
ListFixed set of exact matchesCities: ["New York", "London"]
RegexPattern-based matchingFlight numbers: [A-Z]{2}\d{4}
PrebuiltCommon types (numbers, dates, etc.)datetimeV2, number, email, currency
Pattern.anyVariable-length entities at end of patternBook titles

CLU Development Workflow

Create Language resource → Create CLU project in Language Studio
→ Define intents → Add utterances → Label entities
→ Train → Evaluate (precision/recall/F1) → Deploy → Call prediction API

Prediction API

POST https://<endpoint>.cognitiveservices.azure.com/language/:analyze-conversations
    ?api-version=2023-04-01

{
  "kind": "Conversation",
  "analysisInput": {
    "conversationItem": {
      "text": "Book a flight to Paris next Friday",
      "id": "1", "participantId": "user"
    }
  },
  "parameters": {
    "projectName": "TravelBot",
    "deploymentName": "production"
  }
}

Orchestration Workflow

An Orchestration project routes user input to multiple CLU projects or Q&A knowledge bases. Acts as a dispatcher — ideal for complex bots with multiple domains.

Use case A single bot that can answer HR questions (Q&A KB), book meetings (CLU project A), and order supplies (CLU project B). The Orchestration project picks the right backend.

Bot Framework Integration

CLU integrates with Azure Bot Service and Bot Framework SDK. The bot middleware calls the CLU prediction endpoint, then routes based on the top intent.

  • DirectLine channel — REST-based bot connection for custom clients
  • Channels — Teams, Webchat, Slack, Facebook, etc.
  • Bot Service — hosts and manages the bot; provides channel routing

Azure AI Document Intelligence

Overview

Azure AI Document Intelligence (formerly Form Recognizer) extracts text, key-value pairs, tables, and structured data from documents using AI models.

Prebuilt Models

Read

General OCR — extract text, handwriting, tables from any document. No field extraction.

General Document

Extracts key-value pairs, tables, and text from general documents. No domain specificity.

Layout

Extracts text, tables, selection marks (checkboxes), and structural layout with bounding boxes.

Invoice

Extracts: VendorName, CustomerName, InvoiceId, InvoiceDate, DueDate, SubTotal, Tax, InvoiceTotal, LineItems.

Receipt

Extracts: MerchantName, TransactionDate, TransactionTime, Items, Subtotal, Tax, Tip, Total.

ID Document

Extracts fields from passports and driver's licenses: FirstName, LastName, DateOfBirth, DocumentNumber, etc.

Business Card

Extracts: ContactNames, CompanyName, JobTitles, Emails, Websites, PhoneNumbers, Addresses.

W-2 / Tax Forms

US tax form extraction with prebuilt field mappings.

Custom Models

Custom Template Model

For fixed-layout forms (same layout every time). Requires 5 labeled samples. Uses anchor patterns to locate fields.

Custom Neural Model

For variable layouts (same content, different formatting). Min 10 samples. More flexible, uses deep learning.

Composed Model

Combines multiple custom models into one endpoint. The service auto-routes documents to the best model.

Exam Tip — Model Selection
  • Known, fixed form layout (tax forms, standardized contracts) → Custom Template
  • Variable layout (different invoice designs) → Custom Neural
  • Already looks like an invoice/receipt/ID → Prebuilt
  • Just need all text → Read API

Document Intelligence Studio & Training

  • Upload 5+ labeled documents to Azure Blob Storage
  • Use Document Intelligence Studio to label fields
  • Labels stored as JSON in the same blob container
  • Train via Studio or REST API; deploy to a named model ID

Analyze API Pattern

# Async: Submit then poll
POST /documentintelligence/documentModels/invoice:analyze?api-version=2023-07-31
→ Returns operation-location header URL

GET <operation-location>
→ Poll until status = "succeeded"
→ Response contains extracted fields with values, confidence, bounding boxes

from azure.core.credentials import AzureKeyCredential

    print(invoice.fields["InvoiceTotal"].value_currency.amount)

Knowledge Mining & Azure AI Search

Azure AI Search Overview

Azure AI Search (formerly Cognitive Search) is a cloud search service. It ingests, enriches (via AI skillsets), indexes, and serves documents for full-text search.

Index

The searchable data store. Defines schema: fields, types, analyzers, suggesters, scoring profiles. Like a database table optimized for search.

Indexer

Crawls a data source and populates the index. Can run on a schedule. Applies optional skillset during indexing.

Data Source

Connection definition to the raw data: Azure Blob, SQL DB, Cosmos DB, Table Storage, ADLS Gen2.

Skillset

Pipeline of AI skills (OCR, NER, key phrases, etc.) applied to documents during indexing to enrich them.

AI Enrichment Pipeline

Data Source → Indexer → [Skillset] → Enriched Document → Index
                               ↓
                          Knowledge Store (optional)

Built-in Skills

Merge SkillCombine multiple text fields into one

Persists enriched data from the skillset to Azure Storage for downstream analysis (Power BI, ML, etc.).

Table Projections

Store enrichments as rows in Azure Table Storage. Good for structured, tabular data.

Object Projections

Store enriched documents as JSON in Azure Blob Storage. Full document with all enrichments.

File Projections

Store extracted images/normalized images as files in Blob Storage.

Semantic Search

Semantic search uses deep language models to re-rank results and provide semantic captions and answers. Requires Standard tier or above.

  • Semantic ranking: re-scores BM25 results using ML model understanding context
  • Semantic captions: extracted, highlighted passages that answer the query
  • Semantic answers: direct answers extracted from top results (if query is a question)
POST /indexes/my-index/docs/search?api-version=2023-11-01
{
  "search": "What is the refund policy?",
  "queryType": "semantic",
  "semanticConfiguration": "my-semantic-config",
  "answers": "extractive|count-3",
  "captions": "extractive|highlight-true"
}

Vector Search

Store embeddings as vector fields in the index. At query time, embed the query and find nearest neighbours.

  • Algorithms: HNSW (Hierarchical Navigable Small World — recommended) and exhaustive KNN
  • Use Azure OpenAI Embeddings model to generate vectors
  • Hybrid = keyword BM25 + vector, merged with Reciprocal Rank Fusion (RRF)

Azure OpenAI Service & Generative AI

Azure OpenAI Overview

Azure OpenAI Service provides REST API access to OpenAI's models (GPT-4, GPT-3.5, DALL-E, Whisper, Embeddings) with Azure enterprise features: private networking, compliance, RBAC, and responsible AI guardrails.

GPT-4 / GPT-4o

Most capable chat/completion models. Multimodal (GPT-4 Vision accepts images). Best for complex reasoning, code, analysis.

GPT-3.5 Turbo

Fast and cost-effective chat model. Good for most conversational and completion tasks.

Embeddings

text-embedding-ada-002 / text-embedding-3-large. Converts text to numerical vectors for semantic search and similarity.

DALL-E 3

Generate images from text prompts. Supports 1024×1024, 1792×1024, and 1024×1792 sizes.

Whisper

Audio transcription and translation model. High accuracy across many languages.

Azure OpenAI Deployment

Models are deployed to your Azure OpenAI resource as named deployments. You call your deployment by its custom name, not the model name.

Resource → Deployment → Model One Azure OpenAI resource can host multiple deployments. E.g., resource my-aoai might have deployment gpt4-prod pointing to model gpt-4.

Chat Completions API

from openai import AzureOpenAI

client = AzureOpenAI(
    azure_endpoint="https://<resource>.openai.azure.com",
    api_key="<key>",
    api_version="2024-02-01"
)

response = client.chat.completions.create(
    model="gpt4-prod",   # deployment name
    messages=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user",   "content": "Explain Azure AI Search in 2 sentences."}
    ],
    temperature=0.7,
    max_tokens=500
)
print(response.choices[0].message.content)

Prompt Engineering

Craft inputs to guide model output quality and format without changing model weights.

Zero-shot

No examples. Just a clear instruction. Works for well-understood tasks: "Classify this review as positive or negative: …"

Few-shot

Provide 2–5 input-output examples before the actual query. Steers format and behavior effectively.

Chain of Thought

Ask the model to reason step-by-step before answering. "Think step by step." Improves complex reasoning accuracy.

System Message

Set persona, tone, constraints, and output format via the system role message. Applied to every turn.

Key Parameters
  • temperature (0–2): Higher = more random/creative. Lower = more deterministic. 0 = greedy decoding.
  • top_p (0–1): Nucleus sampling. Use temperature OR top_p, not both.
  • max_tokens: Max tokens in completion (not including prompt).
  • stop: Sequences where the model stops generating.
  • frequency_penalty: Penalizes repeated tokens (reduces repetition).
  • presence_penalty: Penalizes tokens already present (increases topic diversity).

Retrieval-Augmented Generation (RAG)

RAG grounds the model's response in your own data by retrieving relevant chunks and injecting them into the prompt.

# RAG Pattern
User Query
    ↓
1. Embed query → vector
2. Search AI Search index (hybrid) → top-K chunks
3. Build prompt:
   System: "Answer only from the provided context."
   Context: [retrieved chunks]
   User: [original query]
4. Send to GPT-4 → grounded answer
5. Optionally include citations from search results
On Your Data (Built-in RAG)

Azure OpenAI "On Your Data" feature auto-connects to AI Search, Blob Storage, or Cosmos DB. Handled by the service — no custom code for simple use cases.

Custom RAG

Implement with your own pipeline: embed documents → AI Search vector index → retrieval + prompt construction → GPT call. Full control over chunking, retrieval, and prompting.

Fine-tuning

Adapt a base model with your own training data (JSONL format with prompt/completion pairs). Use when prompt engineering alone is insufficient.

Fine-tuning vs RAG Fine-tuning bakes knowledge/style into model weights — expensive to retrain when data changes. RAG is cheaper and keeps knowledge up-to-date. Prefer RAG; use fine-tuning only for style adaptation or highly repetitive task patterns.
ApproachUse whenCost
Prompt engineeringMost cases — start hereOnly inference cost
RAGNeed current/private knowledgeInference + search
Fine-tuningConsistent format/style, insufficient via promptingTraining + higher inference

Azure AI Studio / Azure AI Foundry

Azure AI Foundry (previously AI Studio) is the unified portal for building AI solutions. It provides:

  • Model catalog — explore and deploy OpenAI, Hugging Face, Mistral, and other models
  • Prompt flow — visual LLM workflow builder for RAG and multi-step pipelines
  • Evaluations — measure model accuracy, groundedness, relevance, fluency
  • Safety evaluations — detect harmful content in model outputs
  • Fine-tuning UI — train models without custom code

Responsible AI Principles (Microsoft)

Fairness

AI systems should treat all people fairly, avoid creating or amplifying bias across groups.

Reliability & Safety

Perform reliably and safely under normal and unexpected conditions. Fail gracefully.

Privacy & Security

Respect privacy, protect sensitive data, secure against adversarial attack.

Inclusiveness

Empower everyone, including people with disabilities. Accessible design.

Transparency

Systems should be understandable. Users should know when they're interacting with AI.

Accountability

Humans should remain accountable for AI decisions. Human oversight mechanisms.

Last-Minute Exam Tips
  • Multi-service resource key works for most services but NOT Azure OpenAI — OpenAI needs its own resource.
  • Containers always need to phone home to Azure (Billing + ApiKey env vars).
  • Custom Vision prediction key ≠ training key — two separate keys.
  • CLU replaces LUIS — if you see LUIS in answers, it's likely outdated but still appears on exams.
  • Translator has a GLOBAL endpoint, not per-region — but region header required with multi-service key.
  • Document Intelligence Async: Submit → get operation URL → GET until "succeeded".
  • AI Search Semantic: Not available on Free or Basic tier.
  • Face Identification: Requires Limited Access — can only detect without approval.
  • RAG vs Fine-tuning: Prefer RAG for live/private data; fine-tune only for style/format.
  • Responsible AI: 6 principles — Fairness, Reliability & Safety, Privacy & Security, Inclusiveness, Transparency, Accountability.
