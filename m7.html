Knowledge Mining & Azure AI Search

Key Takeaways:
  • Azure AI Search: ingest, enrich, index, and search documents at scale
  • AI skillsets: chain vision, language, and custom skills for data enrichment
  • Built-in skills: OCR, image analysis, entity recognition, key phrase extraction
  • Full-text search with filters, facets, and scoring profiles
  • Knowledge stores: persist enriched data for further analysis in data lakes

Azure AI Search Overview

Azure AI Search (formerly Cognitive Search) is a cloud search service. It ingests, enriches (via AI skillsets), indexes, and serves documents for full-text search.

Index

The searchable data store. Defines schema: fields, types, analyzers, suggesters, scoring profiles. Like a database table optimized for search.

Indexer

Crawls a data source and populates the index. Can run on a schedule. Applies optional skillset during indexing.

Data Source

Connection definition to the raw data: Azure Blob, SQL DB, Cosmos DB, Table Storage, ADLS Gen2.

Skillset

Pipeline of AI skills (OCR, NER, key phrases, etc.) applied to documents during indexing to enrich them.

AI Enrichment Pipeline

Data Source → Indexer → [Skillset] → Enriched Document → Index
                               ↓
                          Knowledge Store (optional)

Exam Tip — Wikipedia Links (Q30) To generate links to Wikipedia articles as part of AI enrichment, use the Entity Linking Skill (#Microsoft.Skills.Text.V3.EntityLinkingSkill). It outputs entity name, Wikipedia URL, and confidence score. The Entity Recognition skill extracts entity text but does NOT provide Wikipedia URLs.

Custom Skills

Extend the pipeline with your own logic via Azure Function or any HTTPS endpoint.

{
  "@odata.type": "#Microsoft.Skills.Custom.WebApiSkill",
  "uri": "https://myfunction.azurewebsites.net/api/CustomSkill",
  "httpMethod": "POST",
  "inputs": [{ "name": "text", "source": "/document/content" }],
  "outputs": [{ "name": "myOutput", "targetName": "customField" }]
}

Knowledge Store

Persists enriched data from the skillset to Azure Storage for downstream analysis (Power BI, ML, etc.).

Table Projections

Store enrichments as rows in Azure Table Storage. Good for structured, tabular data.

Object Projections

Store enriched documents as JSON in Azure Blob Storage. Full document with all enrichments.

File Projections

Store normalized binary files (extracted images, PDFs) in Azure Blob Storage. Use this for binary content extracted during indexing.

Exam Tip — Binary File Projections (Q52) To save normalized binary files as projections in the knowledge store, use File projections. Table projections are for structured row data; Object projections are for JSON documents.

Skillset Definition — Minimum Required Sections

When creating a skillset via the REST API, the only required field is the skills array. All other fields (name, description, cognitiveServices, knowledgeStore) are optional.

{
  "skills": [
    {
      "@odata.type": "#Microsoft.Skills.Text.V3.EntityLinkingSkill",
      "inputs": [{ "name": "text", "source": "/document/content" }],
      "outputs": [{ "name": "entities", "targetName": "linkedEntities" }]
    }
  ]
}
Exam Tip — Skillset Minimum (Q17) A valid skillset definition requires only the "skills" array. Name and description are optional. The cognitiveServices key is needed only for billable skills; the knowledgeStore block is needed only if you want projections.

Indexing Pipeline Stages

The indexer processes documents through these stages in order:

StageWhat Happens
1. Document CrackingOpens and reads source files — extracts raw text and binary content. This is the first stage.
2. Field MappingMaps source document fields to index fields as defined in the indexer configuration.
3. Skillset ExecutionRuns AI enrichment skills on the extracted content to add new fields (entities, key phrases, captions, etc.).
4. Output Field MappingMaps skill output values to index fields.
5. IndexingWrites the final enriched document to the search index.
Exam Tip — First Indexing Stage (Q42) The first stage of the indexing process is Document Cracking — opening and reading the source document to extract its raw content.

Exam Tip — Wildcard Queries (Q37) Wildcard queries (* and ?) require Lucene query syntax. You must set queryType=full in the REST API request. The default queryType=simple does not support wildcards.

Semantic Search

Semantic search uses deep language models to re-rank results and provide semantic captions and answers. Requires Standard tier or above.

  • Semantic ranking: re-scores BM25 results using ML model understanding context
  • Semantic captions: extracted, highlighted passages that answer the query
  • Semantic answers: direct answers extracted from top results (if query is a question)
POST /indexes/my-index/docs/search?api-version=2023-11-01
{
  "search": "What is the refund policy?",
  "queryType": "semantic",
  "semanticConfiguration": "my-semantic-config",
  "answers": "extractive|count-3",
  "captions": "extractive|highlight-true"
}

Vector Search

Store embeddings as vector fields in the index. At query time, embed the query and find nearest neighbours.

  • Algorithms: HNSW (Hierarchical Navigable Small World — recommended) and exhaustive KNN
  • Use Azure OpenAI Embeddings model to generate vectors
  • Hybrid = keyword BM25 + vector, merged with Reciprocal Rank Fusion (RRF)
